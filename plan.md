# MathBattle 要件・実装計画

## 0. 前提

このプロジェクトは、計算・学習問題をゲーム化するためのテンプレートアプリとして設計する。
UI、ゲームロジック、キャラクター定義、出題ロジックは分離する。

確定済みの大方針を勝手に変更しない。
未確定事項は仮決めとして扱い、あとで差し替え可能にする。

## 1. アプリ概要

- アプリ名：MathBattle
- コンセプト：問題に答えてモンスターを倒す学習バトルアプリ
- 対象：小学生を中心とした学習者
- 目的：算数・学習問題を、RPG風の進行と報酬で継続しやすくする
- 将来用途：別教科・別ゲームにも流用できるテンプレート

  - 復習画面表示

### 2.2 ゲームロジック

- `js/gameLogic.js`
  - バトル状態管理
  - HP管理
  - ダメージ計算
  - コンボ計算
  - パワーゲージ計算
  - 勝敗判定

  - 賞金計算

ゲームロジックにUI描画処理を混ぜない。
ゲームロジックに算数の問題生成処理を混ぜない。

## 3. ゲーム進行

### 3.1 ステージ

- 1ステージは5体のモンスターで構成する。
- 1モンスターは1単元に対応する。
- 1モンスター撃破は、その単元クリアを意味する。
- モンスターを撃破するとバッジを1つ獲得する。
- 同じモンスターから獲得できるバッジは1つだけ。
- 1ステージ内のバッジを5つ集めるとステージクリア。
- ステージクリアすると次ステージが解放される。
- クリア済みステージ、撃破済みモンスターにも再挑戦できる。

### 3.2 ステージオープン

- ステージクリアとステージオープンは別概念とする。
- 100問モードでクリアしていなくても、賞金を貯めることで未クリアの次ステージをオープンできる。
- 賞金でステージをオープンしても、前ステージをクリア済みにするわけではない。
- 賞金でオープンしたステージは挑戦可能になる。
- バッジ・単元クリア・ステージクリアの実績は100問モードでのみ獲得する。

## 4. プレイモード

### 4.1 100問モード

- ステージ進行に使う唯一のモード。
- モンスター撃破、単元クリア、バッジ獲得、ステージクリア判定は100問モードのみで行う。
- 100問以内にボスHPを0にしたら勝利。
- 100問終了時にボスHPが残っていれば敗北。
- 勝利後に正答率やスピードを追加判定して失敗扱いにはしない。
- 正答率とスピードは、HP・ダメージ計算のバランスとして勝敗に反映する。

### 4.2 問題数選択モード

- ユーザーが問題数を選べる。
- 例：10問、20問、50問、100問。
- ステージ進行には使わない。
- バッジ獲得には使わない。
- 単元クリアには使わない。
- 賞金集め、練習、記録更新に使う。
- 問題数に応じて賞金や報酬を調整する。

### 4.3 タイムアタックモード

- 制限時間内にできるだけ多く正解するモード。
- 例：30秒、60秒、120秒、300秒。
- ステージ進行には使わない。
- バッジ獲得には使わない。
- 単元クリアには使わない。
- 賞金集め、スコア挑戦、練習に使う。
- 制限時間、正解数、連続正解数、回答速度に応じて賞金を付与する。

## 5. 勝敗・ダメージ

### 5.1 勝敗

- 勝利：100問以内にボスHPを0にする。
- 敗北：100問終了時にボスHPが残っている。
- 勝利判定はHPのみで行う。
- 正答率・スピードは、HPを削れるかどうかに影響する要素として扱う。

### 5.2 ダメージ

ダメージは以下の要素から計算する。

- 基礎ダメージ
- 正誤
- 回答速度
- 問題ごとの期待回答速度
- コンボ
- パワーゲージ
- モンスター防御力

期待回答速度はゲーム側で固定しない。
期待回答速度は問題設定側で定義する。

### 5.3 コンボ

- 連続正解で増加する。
- 不正解でリセットする。
- ダメージ補正に影響する。
- 表示は一定コンボ以上で行う。
- コンボ数自体は保存してよいが、補正倍率には上限を設ける。

### 5.4 パワーゲージ

- 0〜100%で管理する。
- 正解で増加する。
- 速く正解すると追加で増加する。
- 不正解で減少する。
- ダメージ計算に影響する。
- ゲージ満タン時の必殺技は現時点では基本仕様に含めない。

### 5.5 プレイヤーHP

- 現時点では導入しない。
- 敵の `attack` は現時点では使わない。
- 敵ターン制バトルにはしない。

## 6. 報酬

### 6.1 バッジ

- 100問モードでモンスターを撃破したときのみ獲得する。
- バッジはモンスターごとに1つ固定。
- 同じモンスターを再撃破しても同じバッジは増えない。
- バッジはステージ進行の中心報酬。
- 1ステージ内で5バッジを集めたらステージクリア。

### 6.2 XP

- XPはレベルアップ用の成長報酬。
- XPはステージ解放には使わない。
- XP付与対象：
  - 回答した問題数
  - 正解数
  - モンスター撃破
  - 初回撃破
  - 自己ベスト更新
- 敗北時も少量XPを付与してよい。
- バッジ獲得済みモンスターの再撃破XPは減衰させる。

### 6.3 賞金

- 賞金は練習・問題数選択・タイムアタックで集めるゲーム内報酬。
- 賞金は未クリアステージのオープンに使える。
- 賞金でステージをオープンしても、ステージクリア扱いにはしない。
- 賞金のその他の使い道は今後追加可能。

### 6.4 スター・ランク

- スターは残す。
- スターはステージ解放には使わない。
- スターはやりこみ・熟練度表示に使う。
- ランクは速度評価として使う。
- バッジは進行、スターは熟練度、ランクはスピード評価として役割を分ける。

## 7. レベルアップ

- レベルアップ定義は別ファイルにする。
- `data/level.json` で管理する。
- JSON形式とする。
- レベルはステージ解放条件には使わない。
- レベルは成長表示、称号、報酬演出に使う。
- XPテーブル方式を採用する。
- 計算式だけで自動算出しない。

レベル定義項目：

- `level`
- `requiredTotalXp`
- `title`
- `message`
- `rewards`

## 8. キャラクター追加性

- キャラクターはあとで簡単に追加できるようにする。
- アプリ本体コードを直接編集しなくても追加できる構造にする。
- キャラクター定義は外部JSONで管理する。
- 新しいモンスターは `data/character.json` に追加するだけで基本的に反映できるようにする。
- モンスター、バッジ、所属ステージ、表示情報、ゲーム用パラメータはキャラクター定義側に持たせる。
- キャラクター定義のバリデーションを用意する。
- 不正なキャラクター定義があってもアプリ全体が壊れないようにする。

推奨モンスター項目：

- `id`
- `name`
- `sprite`
- `stageId` または `difficulty`
- `unitId`
- `hp`
- `defense`
- `xpReward`
- `coinReward`
- `origin`
- `description`
- `badge`

## 9. 出題仕様

### 9.1 問題コンテンツ

問題は以下の素材を使えるようにする。

- 文章
- 計算式
- 画像
- 動画
- 音声

問題コンテンツ形式と回答形式は分離する。

組み合わせ例：

- 計算式 × 数値入力
- 文章問題 × 四択
- 画像問題 × 四択
- 音声問題 × 文字入力
- 動画問題 × 四択

### 9.2 回答形式

以下の回答形式に対応する。

- 数値入力
- 四択
- 文字入力

回答形式は独立した部品として扱う。
ゲームロジックに回答UIを直書きしない。
問題定義または単元定義側で、どの回答形式を使うか指定できるようにする。

### 9.3 数値入力

- 計算問題向け。
- 画面テンキーに対応する。
- キーボード入力にも対応できる構造にする。
- 正解判定は数値として行う。

### 9.4 四択

- 計算、用語、図形、文章題向け。
- 選択肢リストを問題データ側で持つ。
- 正解選択肢IDまたは正解値で判定する。

### 9.5 文字入力

- 用語、単位、読み、短い答え向け。
- ひらがな、カタカナ、漢字、英数字に対応できるようにする。
- 表記ゆれは問題定義側の正解定義で管理する。

### 9.6 期待回答速度

- 期待回答速度は問題設定側で決める。
- 問題または単元ごとに期待回答速度を持てるようにする。
- ゲーム側は期待回答速度を参照して、速度評価とダメージ計算に使う。
- ゲーム側が固定の期待回答速度を勝手に持たない。

## 10. 問題画面レイアウト

- 問題画面は、スクロールしなくても全て見れるようにする。
- スマホ画面を基準に設計する。
- 問題本文、素材、回答UI、進捗、タイマー、必要なゲーム情報が1画面内に収まること。
- 画像・動画・音声UIは画面内に収まるサイズに自動調整する。
- 長すぎる文章問題は、問題データ側または表示テンプレート側で短く分割する。
- 回答操作のためにスクロールが必要になる画面にはしない。
- 復習画面や詳細説明画面はスクロール可。
- 本番の問題回答画面はスクロール不可。

## 11. 復習・再テスト

- 回答後に答えを確認できるようにする。
- 結果画面または振り返り画面で、間違った問題を確認できるようにする。
- 間違った問題だけを集めて再テストできるようにする。
- 再テストは通常バトルとは独立した復習モードとして扱う。
- 再テスト対象は、直近バトルの誤答だけでなく、履歴から選べるようにすることを想定する。
- 間違った問題の再テストでは、正答、自分の回答、解き直し結果を記録できるようにする。
- 再テストではボスHPやバッジ進行は基本的に使わない。
- 再テストはXP少量付与または賞金少量付与として扱う。
- 再テストで全問正解した場合は「復習完了」として記録する。

## 12. AI機能

- AI解説は削除する。
- AI解説に関する機能は実装しない。
- AI API呼び出しは実装しない。
- AIによる誤答解説は実装しない。
- AI学習アドバイスは実装しない。
- AI復習問題生成は実装しない。
- 設定画面のAI連携セクションは削除対象。
- API Key入力欄は削除対象。
- プロンプト入力欄は削除対象。
- 既存データにAI設定が残っていても読み込み時に無視する。
- データエクスポートにはAI設定を含めない。

## 追加要件：公開本体から管理用キャラクター設定を削除

### 確定事項

- 公開本体・テンプレートには管理用のキャラクター設定UIを載せない。
- `data/character.json` は実行時データとして読み込む。
- キャラクター追加は、管理者が `data/character.json` を編集してデプロイする運用にする。
- 公開本体では `character.json` のインポート、リセット、編集を行わない。
- キャラクター追加性は維持するが、管理UIは本体から分離する。

## 追加要件：問題設定ツール

### 確定事項

- 問題設定用のツールを追加する。
- 問題設定用ツールはGitHub Pages本番ページには含めない。
- 問題設定用ツールはスマホ端末ローカルでだけ使う。
- ローカル環境で、許可されたGoogleアカウントでログインした場合だけツール本体を表示する。
- 問題を授受するファイル形式を定義する。
- ファイル形式はMathBattle専用に閉じず、汎用的に使える形式にする。
- 問題生成はGeminiを使って実装できるようにする。

### 実装方針

- GitHub Pages公開対象は `github-pages/` のみ。
- 問題設定ツールは `local-tools/problem-maker/` に分離する。
- Google Identity Servicesでログインする。
- `local-tools/problem-maker/config/local-tool.config.js` にGoogle OAuth Client IDと許可メールアドレスを置く。
- 問題パッケージ形式は `learning.problemPackage` v1 とする。
- JSON Schemaは `shared/schemas/problem-package.schema.json` に置く。
- 仕様説明は `shared/problem-format.md` に置く。
- 問題作成ツールで `.learning-pack.json` を作り、公開反映時は `mathUnits.json` に変換する。

### 問題設定ツールの機能

- `.learning-pack.json` をインポートする。
- `.learning-pack.json` をエクスポートする。
- Gemini APIで問題パッケージJSONを生成する。
- 生成JSONを確認・検証する。
- `mathUnits.json` として保存する。

### セキュリティ注意

- ローカルツールのGoogleログインは、ツール画面表示のための制御。
- GitHub Pages公開版にはGoogleログインUIもGemini UIも置かない。
- Gemini API Keyは保存しない。
- ブラウザでGemini API Keyを扱う構成はローカル利用前提。
- 本番で問題作成ツールをオンライン化する場合は、サーバー側でGoogle IDトークン検証とGemini API呼び出しを行う。

## 追加要件：GitHub Pages公開とローカル問題作成ツール

- 本番ページは GitHub Pages で公開する。
- 問題作成用ツールは GitHub Pages 本番ページには含めない。
- 問題作成用ツールはスマホ端末ローカルでだけ使う。
- ローカル問題作成ツールはGoogleログインで許可されたアカウントだけが操作できる。
- Geminiを使って問題パッケージを生成できる。
- 問題授受形式は `learning.problemPackage` v1 / `.learning-pack.json` とする。
- 本番へ反映する場合は、生成済み問題を `mathUnits.json` に変換して `github-pages/data/mathUnits.json` として配置する。

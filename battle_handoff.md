# MathBattle 引き継ぎ文書

作成日: 2026-05-03

## 1. このプロジェクトの現在地

MathBattle は、算数・学習問題をゲーム化するための静的Webアプリです。GitHub Pagesで公開する本体と、スマホ端末ローカルで使う問題作成ツールを分けています。

現在のZIPは、公開用ファイルをルート直下に置き、ローカル保存・問題作成用ツールを `local-save-deploy/` に分離する構成です。

## 2. これまでの経緯

1. 最初に、単一HTMLの計算バトルアプリが提示された。
2. アプリ内容を確認し、小学生向け計算RPG「MathBattle」として仕様を整理した。
3. 未実装箇所を洗い出した。
4. AI解説は不要と判断され、最終的に削除対象になった。
5. ステージアンロック仕様を見直した。
6. レベルアップ定義は本体コードから外し、別ファイル化する方針になった。
7. ゲームロジックと算数・出題ロジックを分離する方針になった。
8. 単元クリアはモンスター撃破と定義した。
9. モンスター撃破は「100問以内にボスHPを0にする」こと。正答率・スピードは追加判定ではなく、HPを削り切るためのバランス条件として扱う方針になった。
10. モンスター撃破でバッジを獲得し、1ステージ内で5バッジを集めるとステージクリアとした。
11. ステージ進行は100問モードだけに限定した。
12. 問題数選択モード、タイムアタック、復習・再テストは賞金集めや練習用とした。
13. 賞金を貯めることで、100問でステージクリアしていなくても次ステージをオープンできるようにした。
14. 回答形式は数値入力、四択、文字入力に対応させる方針になった。
15. 問題素材は文章、計算式、画像、動画、音声に対応する方針になった。
16. 問題画面はスマホでスクロールせずに全体が見えることを要件化した。
17. 間違った問題の答え確認、振り返り、間違いだけ再テストを要件化した。
18. 問題作成ツールを追加し、問題授受形式 `learning.problemPackage` v1 を定義した。
19. 問題作成は Gemini を使えるようにした。ただし公開本体には Gemini UI や API Key 入力欄は置かない。
20. 本番公開は GitHub Pages。問題作成ツールはスマホ端末ローカルでのみ使うことになった。
21. 公開用をルート直下に置く構成へ変更した。
22. 直近の指摘として、公開本体・テンプレートに管理用キャラクター設定を載せない方針になった。
23. 同じく直近の指摘として、ボタンがタップしても動作しない問題があったため、クリック依存を減らし、`pointerup` / `touchend` / `click` に対応する共通タップ処理を追加した。

## 3. 確定要件

### 3.1 公開本体

- GitHub Pagesで公開する。
- 公開用ファイルはZIPルート直下に置く。
- 公開本体には以下を置かない。
  - 問題作成ツール
  - Gemini UI
  - Gemini API Key入力欄
  - GoogleログインUI
  - 管理用キャラクター設定UI
  - `character.json` のインポート/リセットUI
- `data/character.json` は実行時データとして使うが、ユーザーが本体画面から管理するUIは置かない。

### 3.2 ローカル用ツール

- `local-save-deploy/problem-maker/` に置く。
- スマホ端末ローカルで使う。
- GitHub Pages公開対象には含めない。
- 問題作成、Gemini生成、`.learning-pack.json` の読み書き、`mathUnits.json` への変換を担当する。
- Googleログイン許可アカウント制御はローカルツール側だけで使う。

### 3.3 ステージ進行

- ステージ進行に使うのは100問モードのみ。
- 100問以内にボスHPを0にしたらモンスター撃破。
- モンスター撃破 = 単元クリア。
- モンスター撃破でバッジを1つ獲得。
- 1ステージ内でバッジ5つを集めるとステージクリア。
- ステージクリアで次ステージがオープンする。
- 賞金を貯めると、ステージクリアしていなくても次ステージをオープンできる。
- 賞金オープンはステージクリア扱いではない。

### 3.4 プレイモード

- 100問モード: ステージ進行・バッジ獲得用。
- 問題数選択モード: 賞金集め・練習用。
- タイムアタック: 賞金集め・スコア挑戦用。
- 復習・再テスト: 間違い直し用。基本的にバッジ進行には使わない。

### 3.5 回答・問題形式

- 回答形式: 数値入力、四択、文字入力。
- 問題素材: 文章、計算式、画像、動画、音声。
- 回答形式と問題素材は独立させる。
- 期待回答速度は問題設定側で決める。
- ゲーム側は期待回答速度を受け取って、速度評価・ダメージ計算・報酬計算に使う。

### 3.6 画面要件

- 本番の問題回答画面はスクロール不要で全体が見えること。
- 復習や詳細画面はスクロール可。
- スマホ画面を基準にする。

### 3.7 AI関連

- AI解説は削除。
- 公開本体にAI設定欄は置かない。
- 公開本体にGemini API Key入力欄は置かない。
- Geminiはローカル問題作成ツールでのみ使う。

## 5. GitHub Pages公開方法

推奨は GitHub Actions による公開。

`.github/workflows/deploy-pages.yml` は、公開対象を次に限定している。

- `index.html`
- `css/`
- `js/`
- `data/`
- `.nojekyll`

`local-save-deploy/` は公開対象に含めない。

注意: GitHub Pages の「Deploy from branch / root」を直接使うと、`local-save-deploy/` も公開される可能性がある。必ずActionsのワークフローで公開するか、ローカル用フォルダを別リポジトリ/非公開場所に分けること。

## 6. 直近の見直し内容

### 6.1 管理用キャラクター設定を公開本体から削除

削除したもの:

- 設定画面の「キャラクター定義」カード
- `character.json` インポートUI
- 初期 `character.json` に戻すUI
- `app.js` のキャラクターインポート・リセット処理
- 起動時にIndexedDBのカスタムキャラクター定義を優先する処理

残したもの:

- `data/character.json` の読み込み
- `data/defaultData.js` のキャラクターフォールバック
- 図鑑・バッジ表示
- キャラクター定義そのものの外部JSON化

理由:

- キャラクター追加性は必要だが、公開本体・テンプレートに管理UIを載せたくないため。
- キャラクターを変更する場合は開発/管理側で `data/character.json` を編集してデプロイする。

### 6.2 タップ不動作対策

対応したもの:

- `js/ui.js` に `UI.activate(element, handler)` を追加。
- `pointerup` / `touchend` / `click` に対応。
- 二重発火防止のため短時間の重複発火を抑制。
- 主要ボタン、ナビ、ステージカード、モンスターカード、テンキー、四択、結果ボタンに適用。
- `css/styles.css` に `touch-action: manipulation` を追加。
- ローカル問題作成ツールにも同様のタップ処理を追加。
- 起動失敗時にトーストへエラーを出す処理を追加。
- `storage.js` にローカルストレージ使用不可時のメモリフォールバックを追加。

## 7. 確認したこと

- JS構文チェック済み。
- JSON構文チェック済み。
- 公開本体から管理用キャラクター設定UIが消えていることを確認。
- 公開本体にGemini UI、GoogleログインUI、問題作成UIが混ざっていないことを確認。
- `.github/workflows/deploy-pages.yml` が公開対象を限定していることを確認。

制限:

- この実行環境ではブラウザ起動が管理者設定でブロックされたため、実ブラウザでのタップ動作までは検証できていない。
- そのため、タップ不動作対策はコード上の修正と静的確認まで。

## 9. 重要な注意点

- 「キャラクターを後から追加できる」は、公開本体に管理UIを出す意味ではない。
- 公開本体では `data/character.json` を読み込むだけ。
- キャラクター管理は開発・管理作業として行う。
- 問題作成ツールはローカル専用。公開本体に統合しない。
- Gemini API Keyを公開リポジトリに置かない。
- 100問モード以外でバッジやステージクリアを発生させない。
